# 01 — Business Context & User Roles

## Core Business

Our core business is **helping enterprises (clients) find the right engineers**. The recruitment process is primarily driven by recruiters based in India, with the Taiwan team providing support for technical interview stages.

Purpose of this system:

- Improve screening efficiency so recruiters can process more candidates in the same amount of time
- Reduce client satisfaction losses caused by incorrect recommendations
- Accumulate structured candidate data to build a long-term competitive advantage

---

## User Roles

| Role | Location | Primary Responsibilities |
|---|---|---|
| **Recruiter** | India | Collect resumes, upload JDs, send questionnaires, use AI Agent to evaluate candidates, schedule stage timelines, collect client feedback and log it in the system |
| **Technical Interviewer** | Taiwan | Use the system-generated Technical Interview Guide to conduct Stage 2 technical interviews, complete post-interview evaluation reports |
| **Manager** | Taiwan / Remote | Monitor overall recruitment data and pass rates, view business value reports, adjust system parameters and Prompt versions |

---

## Multi-Recruiter Architecture

- The system supports **multiple Recruiters working simultaneously**, each maintaining their own JDs, questionnaire templates, and candidate data independently
- Initially, each Recruiter's data is isolated and not visible to others
- Manager has global report visibility across all Recruiters
- Architecture must support future expansion to "shared questionnaire template library" or "cross-Recruiter candidate visibility"

See [02-core-design.md — Multi-Recruiter Architecture Design](02-core-design.md)

## Glossary

| Term | Description |
|---|---|
| **JD** | Job Description. Describes position requirements, tech stack, and responsibilities. |
| **QA** | In this system, refers to evaluation questionnaire. Auto-generated by the system based on JD. |
| **Resume** | Candidate resume, supports `.docx` or `.pdf` file upload. |
| **Stage 1** | Written questionnaire evaluation phase, driven by India Recruiter. |
| **Stage 2** | Technical interview phase, conducted by Taiwan Interviewer. |
| **Semantic Kernel (SK)** | Microsoft open-source AI application framework for orchestrating LLM calls and plugins. |
| **Azure OpenAI** | OpenAI service on Microsoft Azure; this system uses the GPT-4o model. |
| **ATS** | Applicant Tracking System. Commercial software for managing recruitment workflows such as Greenhouse, Workday, Lever. **This company currently does not use any ATS; this system is the primary recruitment management platform.** |
| **Feedback** | Client (enterprise) responses to recommended candidates. Collected by Recruiter from clients and manually logged into the system. |
| **Talent Pool** | Long-term storage repository for historical candidate data. |
| **Rubric** | Scoring criteria. Defines the scoring basis for each evaluation dimension. |
| **Plugin** | Functional module in Semantic Kernel; each Plugin encapsulates one AI operation (e.g. JD parsing, questionnaire generation). |
