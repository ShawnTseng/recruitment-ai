# 02 — 核心設計理念

## 設計哲學：以「實際經驗分享」取代傳統問答

傳統問卷的核心問題：考知識點，候選人可使用 ChatGPT 等工具生成完美答案。

本系統採用不同策略：

> **要求候選人描述真實做過的專案**，並要求具體細節——時間、團隊規模、技術版本、遇到的問題

AI 工具可以生成通用的「最佳實踐」答案，但無法偽造**候選人個人從未有過的具體記憶**。

---

## 問卷架構：兩層設計

每份問卷由兩層組成，兼顧標準化與靈活性：

```
Layer 1：公版問題
  ↳ 系統根據 JD 自動生成
  ↳ 每個 JD 類別有預設問題集（按技術關鍵字分類）
  ↳ 持續根據客戶 Feedback 迭代優化

Layer 2：Recruiter 客製化
  ↳ Recruiter 可在公版基礎上新增 / 修改 / 刪除問題
  ↳ 可調整追問深度（Basic / Advanced / Deep）
  ↳ 儲存為個人專屬範本供日後重用
  ↳ （未來）可選擇性將優秀範本升級為公用範本
```

---

## 防 AI 作答機制

| 策略 | 具體做法 |
|---|---|
| **追問具體細節** | Agent 對模糊答案自動追問（例：「你說你負責架構設計，當時團隊幾人？為什麼選這個方案？」） |
| **時間與版本錨定** | 要求候選人說明使用的技術版本與時間點，AI 難以生成時間軸一致的回答 |
| **履歷交叉比對** | Agent 將履歷聲明與問卷回答交叉比對，標記前後不一致處 |
| **「踩過什麼坑」框架** | 問題問失敗與教訓而非最佳實踐，AI 無法偽造個人踩坑記憶 |
| **AI 識別警示聲明** | 問卷開頭明確告知：通用 AI 生成答案將被自動識別並調降評分 |
| **候選人知情同意** | 問卷開頭聲明本系統使用 AI 輔助評估，候選人需勾選同意後才能作答 |

## 多 Recruiter 架構設計

### 當前階段：獨立模式

- 每位 Recruiter 有獨立的工作空間（JD 清單、問卷範本、候選人資料）
- 資料以 `workspace_id` 隔離，API 層強制驗證
- Manager 擁有跨所有 Recruiter 的唯讀全域視角

### 未來可擴展：共用模式

| 功能 | 說明 |
|---|---|
| 公用問卷範本庫 | Recruiter 可發布優秀範本供他人使用 |
| 跨 Recruiter 候選人查重 | 避免同一候選人被不同 Recruiter 重複評估 |
| 角色分層 | Team Lead Recruiter 可查看本組匯總數據 |

## 版本控管與持續改善循環

```
新 JD → 生成 QA → 候選人作答 → AI 評估 → 人工確認
                                              ↓
                                 記錄評估結果（通過 / 拒絕）
                                              ↓
                              客戶 Feedback → Recruiter 登錄
                                              ↓
                         Feedback Loop Plugin 分析模式
                                              ↓
               自動建議更新 Prompt 版本 → Manager 審核 → 套用
```

> 每份 JD 的問卷保有版本記錄，可回溯比較不同版本的通過率與品質效果（A/B 測試）。

## 評分 Rubric 標準化

每道題依三個維度評分，讓 AI 輸出具備一致性與可解釋性：

| 維度 | 說明 | 預設權重 |
|---|---|---|
| Technical Depth | 回答展現的技術理解深度 | 50% |
| Specificity | 細節具體程度（版本、時間、規模） | 30% |
| Relevance to JD | 與 JD 要求的相關度 | 20% |

評分等級：**Strong / Acceptable / Needs Probe / Insufficient**

> 各維度權重可由 Manager 在系統參數設定中調整。詳見 [05-manager-dashboard.md](05-manager-dashboard.md)

---

## Human-in-the-Loop 原則

AI 評估結果僅作為輔助——**每個關卡的最終決定由人工確認**：

- Stage 1：Recruiter 審閱 AI 報告後決定是否進入 Stage 2
- Stage 2：Interviewer 填寫面試後回饋，覆蓋或確認 AI 建議
- 系統標記 AI 與人工決定不一致的案例，作為模型改善的依據
