# 02 — Core Design Philosophy

## Design Philosophy: "Real Experience Sharing" over Traditional Q&A

The core problem with traditional questionnaires: they test knowledge points, and candidates can use tools like ChatGPT to generate perfect answers.

This system takes a different approach:

> **Require candidates to describe real projects they have actually worked on**, with specific details — timeline, team size, technology versions, and challenges encountered

AI tools can generate generic "best practices" answers, but cannot fabricate **specific memories a candidate has never personally had**.

---

## Questionnaire Structure: Two-Layer Design

Each questionnaire consists of two layers, balancing standardization with flexibility:

```
Layer 1: Public Questions
  ↳ Auto-generated by the system based on JD
  ↳ Each JD category has a preset question set (categorized by tech keywords)
  ↳ Continuously iterated based on client feedback

Layer 2: Recruiter Customization
  ↳ Recruiter can add / modify / delete questions on top of public base
  ↳ Can adjust follow-up depth (Basic / Advanced / Deep)
  ↳ Saved as personal templates for future reuse
  ↳ (Future) Option to promote excellent templates to shared public templates
```

---

## Anti-AI-Cheating Mechanisms

> **Process note**: Candidate questionnaire answers are **first collected by the Recruiter** then uploaded to the system for AI Agent evaluation. AI does not interact directly with candidates and has no automatic follow-up function. Follow-up logic is embedded in the questionnaire design (multi-layer question structure).

| Strategy | Implementation |
|---|---|
| **Multi-layer question design** | Questionnaire pre-designs follow-up layers (Basic / Advanced / Deep), requiring candidates to answer details in the same form — e.g. "You said you were responsible for architecture design, describe the team size, technology selection rationale, and challenges encountered" |
| **Time and version anchoring** | Require candidates to state the technology version and time period used; AI struggles to generate temporally consistent answers |
| **Resume cross-validation** | Agent cross-references resume claims with questionnaire answers, flagging inconsistencies |
| **"Lessons learned" framing** | Questions ask about failures and lessons rather than best practices; AI cannot fabricate personal failure memories |
| **AI detection warning** | Questionnaire header explicitly states: generic AI-generated answers will be automatically identified and scores reduced |
| **Candidate informed consent** | Questionnaire header declares that this system uses AI-assisted evaluation; candidates must check a consent box before answering |

## User Roles & Account Management

### Role Definitions

| Role | Description | Access Scope |
|---|---|---|
| **Super Admin** | Super administrator (e.g. system owner) | All features, all pages, all Recruiter workspaces |
| **Manager** | Administrator | Cross-Recruiter global read-only view, system parameter management, feedback management, report export |
| **Recruiter** | India-based recruiter | Own workspace only (JDs, questionnaires, candidates) |
| **Technical Interviewer** | Taiwan technical interviewer | Assigned interview tasks only |
| **Account Manager** | Business person managing client relationships | Log client feedback (limited to related candidates) |

### Authentication Design

This system uses **Azure Entra ID (external identity provider)** for authentication — no need to manage passwords:

- Login portal: One-click login with organization Microsoft account (company email), no extra passwords to remember
- Adding users: Manager or Super Admin creates accounts in the backend and assigns roles and workspaces
- Role authorization: JWT Token contains role claims; API layer validates with `[Authorize(Roles = "...")]`
- Super Admin is not restricted by `workspace_id` isolation and can access all data

> **Why not build our own username/password system?** Azure Entra ID eliminates the security complexity of password storage, encryption, and reset, and organization members already have existing accounts — just invite them to join. Ideal choice for internal systems.

### Multi-Recruiter Workspace Isolation

- Each Recruiter has an independent workspace (JD list, questionnaire templates, candidate data)
- Data is isolated by `workspace_id`, enforced at the **Repository layer** (not just the controller layer)
- Manager and Super Admin have global cross-workspace visibility

### Future Extensibility: Shared Mode

| Feature | Description |
|---|---|
| Shared questionnaire template library | Recruiters can publish excellent templates for others to use |
| Cross-Recruiter candidate deduplication | Prevent the same candidate from being evaluated by different Recruiters |
| Role hierarchy | Team Lead Recruiter can view aggregated data for their team |

## Version Control & Continuous Improvement Loop

```
New JD → Generate QA → Candidate answers → Recruiter uploads answers → AI evaluation → Human review
                                                                                   ↓
                                                                    Record evaluation result (Pass / Reject)
                                                                                   ↓
                                      Client Feedback → Recruiter / Account Manager logs it
                                                                                   ↓
                                              Feedback Loop Plugin analyzes patterns
                                                                        ↓
                             Auto-suggest updated Prompt version → Manager reviews → Apply
```

> Each JD's questionnaire maintains version history, allowing retrospective comparison of pass rates and quality across different versions (A/B testing).

## Scoring Rubric Standardization

Each question is scored on three dimensions, giving AI output consistency and explainability:

| Dimension | Description | Default Weight |
|---|---|---|
| Technical Depth | Depth of technical understanding demonstrated in the answer | 50% |
| Specificity | Level of detail (versions, timelines, scale) | 30% |
| Relevance to JD | Relevance to JD requirements | 20% |

Scoring levels: **Strong / Acceptable / Needs Probe / Insufficient**

> Dimension weights can be adjusted by Manager in system parameter settings. See [05-manager-dashboard.md](05-manager-dashboard.md)

---

## Human-in-the-Loop Principle

AI evaluation results serve only as assistance — **the final decision at each stage is confirmed by a human**:

- Stage 1: Recruiter reviews AI report and decides whether to proceed to Stage 2
- Stage 2: Interviewer fills in post-interview feedback, overriding or confirming AI suggestions
- The system flags cases where AI and human decisions are inconsistent, as input for model improvement
