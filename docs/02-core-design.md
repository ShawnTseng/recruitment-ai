# 02 — 核心設計理念

## 設計哲學：以「實際經驗分享」取代傳統問答

傳統問卷的核心問題：考知識點，候選人可使用 ChatGPT 等工具生成完美答案。

本系統採用不同策略：

> **要求候選人描述真實做過的專案**，並要求具體細節——時間、團隊規模、技術版本、遇到的問題

AI 工具可以生成通用的「最佳實踐」答案，但無法偽造**候選人個人從未有過的具體記憶**。

---

## 問卷架構：兩層設計

每份問卷由兩層組成，兼顧標準化與靈活性：

```
Layer 1：公版問題
  ↳ 系統根據 JD 自動生成
  ↳ 每個 JD 類別有預設問題集（按技術關鍵字分類）
  ↳ 持續根據客戶 Feedback 迭代優化

Layer 2：Recruiter 客製化
  ↳ Recruiter 可在公版基礎上新增 / 修改 / 刪除問題
  ↳ 可調整追問深度（Basic / Advanced / Deep）
  ↳ 儲存為個人專屬範本供日後重用
  ↳ （未來）可選擇性將優秀範本升級為公用範本
```

---

## 防 AI 作答機制

> **流程說明**：候選人的問卷回答由 **Recruiter 收集後** 上傳至系統，再交由 AI Agent 評估。AI 不直接與候選人互動，無自動追問功能。追問邏輯內嵌於問卷設計中（多層次問題結構）。

| 策略 | 具體做法 |
|---|---|
| **多層次問題設計** | 問卷預先設計追問層次（Basic / Advanced / Deep），要求候選人在同一表單中回答細節，例如：「你說負責架構設計，請說明當時團隊規模、選型原因與遇到的挑戰」 |
| **時間與版本錨定** | 要求候選人說明使用的技術版本與時間點，AI 難以生成時間軸一致的回答 |
| **履歷交叉比對** | Agent 將履歷聲明與問卷回答交叉比對，標記前後不一致處 |
| **「踩過什麼坑」框架** | 問題問失敗與教訓而非最佳實踐，AI 無法偽造個人踩坑記憶 |
| **AI 識別警示聲明** | 問卷開頭明確告知：通用 AI 生成答案將被自動識別並調降評分 |
| **候選人知情同意** | 問卷開頭聲明本系統使用 AI 輔助評估，候選人需勾選同意後才能作答 |

## 使用者角色與帳號管理

### 角色定義

| 角色 | 說明 | 可存取範圍 |
|---|---|---|
| **Super Admin** | 超級管理者（例：系統擁有者） | 所有功能、所有頁面、所有 Recruiter 工作空間 |
| **Manager** | 管理者 | 跨 Recruiter 全域唯讀視角、系統參數管理、Feedback 管理、報告匯出 |
| **Recruiter** | 印度端招募人員 | 僅限自己的工作空間（JD、問卷、候選人） |
| **Technical Interviewer** | 台灣技術面試官 | 指派給自己的面試任務 |
| **Account Manager** | 負責客戶關係的業務人員 | 登錄客戶 Feedback（僅限相關候選人） |

### 身份驗證設計

本系統使用 **Azure Entra ID（外部識別提供者）** 進行身份驗證，無需自行管理帳號密碼：

- 登入入口：使用組織 Microsoft 帳號（公司 Email）一鍵登入，無需記憶額外密碼
- 新增使用者：Manager 或 Super Admin 在後台建立帳號，並指派角色與工作空間
- 角色授權：JWT Token 中包含角色聲明（Claims），API 層以 `[Authorize(Roles = "...")]` 驗證
- Super Admin 不受 `workspace_id` 隔離限制，可存取所有資料

> **為何不用自建帳號密碼？** Azure Entra ID 免去密碼儲存、加密、重設等安全複雜度，且組織成員已有現成帳號，只需邀請加入即可。適合內部系統首選。

### 多 Recruiter 工作空間隔離

- 每位 Recruiter 有獨立的工作空間（JD 清單、問卷範本、候選人資料）
- 資料以 `workspace_id` 隔離，**Repository 層**強制過濾（非僅控制器層）
- Manager 與 Super Admin 擁有跨工作空間的全域視角

### 未來可擴展：共用模式

| 功能 | 說明 |
|---|---|
| 公用問卷範本庫 | Recruiter 可發布優秀範本供他人使用 |
| 跨 Recruiter 候選人查重 | 避免同一候選人被不同 Recruiter 重複評估 |
| 角色分層 | Team Lead Recruiter 可查看本組匯總數據 |

## 版本控管與持續改善循環

```
新 JD → 生成 QA → 候選人作答 → Recruiter 上傳回答 → AI 評估 → 人工確認
                                                                  ↓
                                                     記錄評估結果（通過 / 拒絕）
                                                                  ↓
                              客戶 Feedback → Recruiter / Account Manager 登錄
                                                                  ↓
                         Feedback Loop Plugin 分析模式
                                              ↓
               自動建議更新 Prompt 版本 → Manager 審核 → 套用
```

> 每份 JD 的問卷保有版本記錄，可回溯比較不同版本的通過率與品質效果（A/B 測試）。

## 評分 Rubric 標準化

每道題依三個維度評分，讓 AI 輸出具備一致性與可解釋性：

| 維度 | 說明 | 預設權重 |
|---|---|---|
| Technical Depth | 回答展現的技術理解深度 | 50% |
| Specificity | 細節具體程度（版本、時間、規模） | 30% |
| Relevance to JD | 與 JD 要求的相關度 | 20% |

評分等級：**Strong / Acceptable / Needs Probe / Insufficient**

> 各維度權重可由 Manager 在系統參數設定中調整。詳見 [05-manager-dashboard.md](05-manager-dashboard.md)

---

## Human-in-the-Loop 原則

AI 評估結果僅作為輔助——**每個關卡的最終決定由人工確認**：

- Stage 1：Recruiter 審閱 AI 報告後決定是否進入 Stage 2
- Stage 2：Interviewer 填寫面試後回饋，覆蓋或確認 AI 建議
- 系統標記 AI 與人工決定不一致的案例，作為模型改善的依據
